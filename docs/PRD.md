# 孙颖莎每日资讯小程序 - 产品需求文档 (PRD) v3.0

## 1. 文档概述

### 1.1 产品背景
为孙颖莎（莎莎）的粉丝群体提供一个自动化、实时的资讯聚合平台。通过技术手段自动抓取全网相关新闻，并利用 AI 进行智能提炼，为用户提供高效、纯净的阅读体验。本项目致力于打造一个**零成本运维**的开源公益项目。

## 2. 核心业务流程

```mermaid
graph TD
    Trigger[GitHub Actions 定时触发] -->|运行爬虫脚本| Crawler(数据采集服务)
    Crawler -->|获取原始网页| Source[全网数据源: 微博/百度/等]
    Crawler -->|原始文本| AI_Service{AI 处理服务}
    AI_Service -->|生成| Content[标题 + 300字图文摘要]
    Content -->|存入| DB[(Supabase 数据库)]
    User[用户] -->|打开小程序| Frontend[Vercel 部署的前端/API]
    Frontend -->|读取| DB
    Frontend -->|展示| UI[资讯列表 (点击展开摘要)]
```

## 3. 功能需求详解

### 3.1 数据采集与 AI 处理 (Backend & AI)

#### 3.1.1 运行环境与调度
- **平台**：GitHub Actions。
- **频次**：
    - 每日定时运行 (如：早8点、午12点、晚8点)。
    - 重大赛事期间可适当增加频次。
- **成本控制**：利用 GitHub Actions 的免费额度 (每月 2000 分钟)。

#### 3.1.2 抓取与 AI 处理逻辑
1.  **抓取**：
    - 爬虫访问目标数据源（微博超话、百度新闻等）。
    - 提取正文、图片链接、发布时间、来源。
2.  **AI 提炼 (核心)**：
    - 调用 LLM API (推荐 DeepSeek, Gemini Flash 等免费/低成本方案)。
    - **Prompt 要求**：
        - 输入：原始新闻/博文内容。
        - 输出 JSON 格式：
            - `title`: 简练的标题 (20字以内)。
            - `summary`: 300字以内的**图文摘要**。保留核心信息、比赛结果、关键言论。若原文有图，需在摘要中穿插 Markdown 格式的图片链接 `![img](url)`。
    - **清洗**：去除广告、无关话题、负面情绪内容。
3.  **存储**：
    - 将处理后的数据直接写入 Supabase 数据库。
    - 字段去重：基于 `source_url` 判断是否已存在。

### 3.2 前端交互 (Frontend)

#### 3.2.1 资讯列表页 (首页)
- **布局**：垂直列表，按 `publish_time` 倒序排列。
- **列表项状态**：
    - **默认状态 (收起)**：
        - 仅展示 **标题**。
        - 显示发布时间 (如 "2小时前") 和来源。
        - 视觉上紧凑，一屏可展示多条。
    - **展开状态 (点击标题后)**：
        - 在标题下方**原地展开**显示 `summary` (图文摘要)。
        - 摘要支持富文本/Markdown 渲染 (文字 + 图片)。
        - 再次点击标题区域，收起摘要。
- **交互细节**：
    - 支持手风琴效果 (Accordion)：允许同时展开多项，或每次只展开一项 (建议允许同时展开)。
    - 加载更多：触底自动加载下一页数据。

#### 3.2.2 移除功能
- **移除详情页**：不再跳转到独立的详情页面，所有内容在列表页通过展开查看。
- **移除外部跳转**：除非摘要中包含必须的视频链接，否则尽量让用户在小程序内完成阅读。

## 4. 数据结构 (Schema)

推荐使用 Supabase (PostgreSQL) 的 `news` 表：

| 字段名 | 类型 | 说明 | 备注 |
| :--- | :--- | :--- | :--- |
| `id` | uuid | 主键 | 默认生成 |
| `created_at` | timestamptz | 创建时间 | 默认 now() |
| `title` | text | 新闻标题 | **AI 生成**，用于列表展示 |
| `summary` | text | 图文摘要 | **AI 生成**，Markdown 格式，含图片链接，< 300字 |
| `publish_time` | timestamptz | 原始发布时间 | 用于排序 |
| `source` | text | 来源 | e.g., "微博", "百度新闻" |
| `source_url` | text | 原始链接 | 用于去重和溯源 |
| `images` | text[] | 图片链接数组 | 原始图片列表 (备份用) |
| `status` | text | 状态 | 'published', 'pending' (默认 published) |

## 5. 技术架构 (全免费方案)

### 5.1 选型推荐
- **数据库 & Auth**: **Supabase** (Free Tier: 500MB database, 足够存储数万条文本资讯)。
- **后端/API**: **Vercel Serverless Functions** (Next.js API Routes) 或直接由前端调用 Supabase SDK。
- **前端托管**: **Vercel** (Free Tier)。
- **定时任务/爬虫**: **GitHub Actions** (Free Tier)。
- **AI 服务**:
    - 方案 A: **Google Gemini API** (Free Tier 可用，需注意配额)。
    - 方案 B: **DeepSeek API** (价格极低，接近免费)。
    - 方案 C: **Coze/Dify** (利用其 Workflow 封装 API)。

### 5.2 架构图
```
[GitHub Actions] --(定时运行)--> [Python/Node 爬虫脚本]
                                    |
                                    v
                                [AI API (提炼摘要)]
                                    |
                                    v
                                [Supabase DB]
                                    ^
                                    | (读写)
                                    |
[用户手机] <--(HTTPS)--> [Vercel (前端/小程序)]
```

## 6. 风险与限制
1.  **API 配额**：免费版 AI API 通常有速率限制 (RPM)，爬虫脚本需增加延时 (`sleep`)，避免并发过高。
2.  **图片存储**：建议直接使用原网站图片链接 (热链)，不转存到 Supabase Storage 以节省流量和存储空间。需处理防盗链问题 (前端使用 `no-referrer` 策略)。
3.  **长效维护**：若原网站改版，爬虫脚本需手动更新。
